\subsection{Technologieauswahl}
Der MVP wird in der Programmiersprache PHP mittels dem Framework Laravel erstellt. 
Im Gegensatz zu .NET (Sprache C Sharp) oder Java Spring ist Laravel deutlich einfacher in Erstellung und Konfigurierung.
Ebenso besitzt Laravel weniger Boilerplate Code, also Code der in verschiedenen Kontexten wiederverwendet werden kann bzw. teilweise muss. \parencite{boiler} 
Eine weitere Option waere die Nutzung der Sprache Javascript mittels Node.JS. 
Node.JS besitzt ein sehr grosses Oekosystem, was wiederum dazu fuehrt, dass jede Funktionalitaet in einer separaten Bibliothek zu finden ist. 
Man muss sozusagen mit vielen Dependencies arbeiten.
Laravel bringt hingegen alle noetigen Features, wie Authentifikation, Routing, Dateiuploads, out of the box mit. 
Als Datenbank wird eine Postgre SQL Database genutzt. 
Dies haengt vor allem mit der Infrastrukturentscheidung im naechsten Punkt zusammen. 
Alternativ waere fuer ein MVP auch eine SQLite Datenbank denkbar, da diese sehr leicht zu implementieren und zu konfigurieren ist. 
Jedoch wuerde dies nicht mit der Infrastrukturentscheidung, die Anwendung via Paas zu hosten, funktionieren, da lokale Daten taeglich geloescht werden.

\subsection{Infrastruktur}
Die Infrastruktur des Systems ist in drei Umgebungen unterteilt: Entwicklungs-, Test- und Produktivumgebung. 
Die Entwicklung erfolgt auf der lokalen Entwicklermaschine.
Die Testumgebung, sowie eine moegliche Produmgebung stellt der Cloud-Hosting Betreiber Heroku zur Verfuegung. 
Es wird der Cloud Service "Platform as a Service", kurz Paas, von Heroku genutzt, da somit das Haupaugenmerk auf die Entwicklung gelegt werden kann und nicht auf die Infrastruktur.
Beim Betreiben von Websites ist heutzutage Cybersecurity ein extrem wichtiger Aspekt.
Da in der ATS Applikation sensible Daten gespeichert werden, ist somit hier hoechste Vorsicht geboten.
Die Absicherung der Infrastruktur bspw. durch Abfangen von DDOS-Attacken, obliegt somit dem Cloud Anbieter. 
Wuerde man bspw. einen eigenen Web-Server betreiben, muesste man sich um die Security selber kuemmern. 
Heroku bietet weiterfuehrend native Unterstuetzung fuer Laravel Anwendungen und Postgre Datenbanken an. 
Dies ist unter Anderem auch der Grund fuer die Nutzung von Postgre als Datenbank. 
Wie bei Cloud-Hosting ueblich, gibt es verschiedene Preismodelle, die fuer die Nutzung erhoben werden. 
Im Falle von Heroku nehmen wir die einfachsten Basic Optionen, weil diese fuer den MVP voellig ausreichend sind. 
Diese sind:
\begin{itemize}
	\item App-Hosting - Basic Paket - 7 Euro \parencite{herokuPricing}
	\item Datenbank-Hosting - Essential Zero - 5 Euro \parencite{herokuPricing}
	\item Monatlich Gesamt - 12 Euro
\end{itemize}

\subsection{Softwarearchitektur}
MVC ist die definierte Architektur fuer die Applikation und steht fuer Model-View-Controller.
Das Model beinhaltet Datenbank- und Businesslogiken, die View stellt das Layout bzw. die UI dar und der Controller bildet die Schnittstelle zwischen Model und View. \parencite{mvc}
Diese Architektur bietet sich fuer kleinere, schlanke Anwendungen an und Laravel bringt diese standardmaessig mit. 
Somit bleibt die technische Umsetzung schlank und angemessen für den Umfang eines Minimum Viable Product.
Fuer groessere und komplexerer Business-Anwendungen finden eher MVVM oder Microservice-Architektur Anwendung.
