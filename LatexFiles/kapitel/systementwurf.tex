\subsection{Datenmodell}
Das Datenmodell wurde mithilfe eines Entity-Relationship-Models erstellt. (vgl. \ref{fig:erm}) 
Die Datentypen der einzelnen Spalten wurden mithilfe der Postgre Dokumentation \parencite{postgreDt} ermittelt, sodass eine Implementierung ohne weitere Konvertierungen erfolgen kann.
Tabellenrelationen, wie \textit{User-Bewerber oder Stelle-Images} besitzen eine 1:1 Beziehung. 
Dass bedeutet, dass genau ein Wert aus einer Tabelle genau einmal in der anderen zu finden ist. 
Ebenso haben wir auch 1:n Beziehungen, bspw. zwischen den Tabellen \textit{Bewerber-Bewerbung}. 
1:n bedeutet in diesem Fall, dass ein Bewerber n-Bewerbungen abgeben kann, jedoch eine Bewerbung genau einem Bewerber zugeordnet ist. 
Ausserdem werden aussergewoehnlichere Datentypen, wie \textit{bytea} fuer PDF Speicherung oder \textit{json} fuer Speicherung von Datensaetzen in Form einer json, genutzt. 

\subsection{Technologieauswahl}
Der MVP wird in der Programmiersprache PHP mittels dem Framework Laravel erstellt. 
Im Gegensatz zu .NET (Sprache C\#) oder Java Spring ist Laravel deutlich einfacher in Erstellung und Konfigurierung.
Ebenso besitzt Laravel weniger Boilerplate Code, also Code der in verschiedenen Kontexten wiederverwendet werden kann bzw. teilweise muss. \parencite{boiler} 
Eine weitere Option waere die Nutzung der Sprache Javascript mittels Node.JS. 
Node.JS besitzt ein sehr grosses Oekosystem, was wiederum dazu fuehrt, dass jede Funktionalitaet in einer separaten Bibliothek zu finden ist. 
Man muss sozusagen mit vielen Dependencies arbeiten.
Laravel bringt hingegen alle noetigen Features, wie Authentifikation, Routing, Dateiuploads, out of the box mit. 
Als Datenbank wird eine Postgre SQL Database genutzt. 
Dies haengt vor allem mit der Infrastrukturentscheidung im naechsten Punkt zusammen. 
Alternativ waere fuer ein MVP auch eine SQLite Datenbank denkbar, da diese sehr leicht zu implementieren und zu konfigurieren ist. 
Jedoch wuerde dies nicht mit der Infrastrukturentscheidung, die Anwendung via Paas zu hosten, funktionieren, da lokale Daten taeglich geloescht werden.

\subsection{Infrastruktur}
Die Infrastruktur des Systems ist in drei Umgebungen unterteilt: Entwicklungs-, Test- und Produktivumgebung. 
Die Entwicklung erfolgt auf der lokalen Entwicklermaschine. 
Die Test- sowie eine mögliche Produktivumgebung stellt der Cloud-Hosting-Betreiber \textit{Laravel Cloud} zur Verfügung \parencite{laravelCloud}.
Es wird der Cloud Service \textit{Platform as a Service}, kurz PaaS, von Laravel Cloud genutzt, da somit das Hauptaugenmerk auf die Entwicklung gelegt werden kann und nicht auf die Infrastruktur.
Beim Betreiben von Websites ist heutzutage Cybersecurity ein extrem wichtiger Aspekt. 
Da in der ATS-Applikation sensible Daten gespeichert werden, ist hier höchste Vorsicht geboten. 
Die Absicherung der Infrastruktur, beispielsweise das Abfangen von DDoS-Attacken, obliegt somit dem Cloud-Anbieter. 
Würde man hingegen einen eigenen Webserver betreiben, müsste man sich um die Security selbst kümmern.
Laravel Cloud wurde speziell für Laravel-Anwendungen entwickelt und bietet native Unterstützung für PostgreSQL-Datenbanken \parencite{laravelCloudDocs}. 
Dies ist auch der Grund für die Nutzung von PostgreSQL als Datenbanksystem.
Wie bei Cloud-Hosting üblich, gibt es verschiedene Preismodelle. 
Laravel Cloud verfolgt ein nutzungsbasiertes Abrechnungsmodell ohne feste monatliche Grundgebühr \parencite{laravelCloudPricing}. 
Die Kosten entstehen ausschließlich durch aktive Nutzung. 
Aufgrund von Auto-Hibernation skaliert die Anwendung bei Inaktivität automatisch auf null herunter, wodurch keine Kosten für Leerlaufzeiten anfallen. 
Für den MVP wird der \textit{Starter Plan} genutzt, welcher folgende Leistungen umfasst:
\begin{itemize}
\item App-Hosting - Starter Plan (Flex Compute, mit Hibernation) \parencite{laravelCloudPricing}
\item Datenbank-Hosting - Serverless PostgreSQL (mit Hibernation) \parencite{laravelCloudPricing}
\item Monatliche Kosten - nutzungsabhängig, ca. 1--2 USD bei geringer Aktivität
\end{itemize}

\subsection{Softwarearchitektur}
MVC ist die definierte Architektur fuer die Applikation und steht fuer Model-View-Controller.
Das Model beinhaltet Datenbank- und Businesslogiken, die View stellt das Layout bzw. die UI dar und der Controller bildet die Schnittstelle zwischen Model und View. \parencite{mvc}
Diese Architektur bietet sich fuer kleinere, schlanke Anwendungen an und Laravel bringt diese standardmaessig mit. 
Somit bleibt die technische Umsetzung schlank und angemessen für den Umfang eines Minimum Viable Product.
Fuer groessere und komplexerer Business-Anwendungen finden eher MVVM oder Microservice-Architektur Anwendung.

\end{itemize}
