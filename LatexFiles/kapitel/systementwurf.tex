\subsection{Datenmodell}
Das Datenmodell wurde mithilfe eines Entity-Relationship-Modells erstellt (vgl. \ref{fig:erm}). 
Die Datentypen der einzelnen Spalten wurden auf Basis der PostgreSQL-Dokumentation \parencite{postgreDt} gewählt, um eine direkte und konsistente Implementierung ohne zusätzliche Konvertierungen zu ermöglichen.
\newline
Zwischen den Tabellen bestehen unterschiedliche Relationen. 
Beispielsweise liegt zwischen den Tabellen \textit{User} und \textit{Bewerber} sowie \textit{Stelle} und \textit{Images} jeweils eine 1:1-Beziehung vor. 
Dies bedeutet, dass jedem Datensatz genau ein entsprechender Datensatz in der jeweils anderen Tabelle zugeordnet ist. 
Darüber hinaus existieren auch 1:n-Beziehungen, beispielsweise zwischen den Tabellen \textit{Bewerber} und \textit{Bewerbung}. 
In diesem Fall kann ein Bewerber mehrere Bewerbungen einreichen, während jede Bewerbung genau einem Bewerber zugeordnet ist.
\newline
Zusätzlich werden spezielle Datentypen wie \textit{bytea} zur Speicherung von PDF-Dokumenten sowie \textit{json} zur strukturierten Ablage komplexer Daten verwendet.

\subsection{Technologieauswahl}
Der MVP wird in der Programmiersprache PHP unter Verwendung des Frameworks Laravel umgesetzt. 
Im Vergleich zu Alternativen wie .NET (C\#) oder Spring (Java) zeichnet sich Laravel durch eine geringere Komplexität in Einrichtung und Konfiguration aus. 
Zudem erfordert Laravel weniger Boilerplate-Code, also wiederkehrenden Standardcode, der in verschiedenen Kontexten erneut implementiert werden muss \parencite{boiler}.
\newline
Eine weitere mögliche Alternative wäre die Nutzung von JavaScript in Kombination mit Node.js. 
Dieses bietet ein umfangreiches Ökosystem, führt jedoch häufig dazu, dass viele Funktionalitäten über externe Bibliotheken integriert werden müssen, wodurch die Anzahl an Abhängigkeiten steigt. 
Laravel hingegen stellt zentrale Funktionen wie Authentifizierung, Routing und Dateiuploads bereits integriert zur Verfügung und ermöglicht somit eine strukturierte und effiziente Entwicklung.
\newline
Als Datenbanksystem wird PostgreSQL eingesetzt. 
Diese Entscheidung steht im Zusammenhang mit der gewählten Infrastruktur, insbesondere im Hinblick auf den Einsatz einer Platform-as-a-Service-Umgebung. 
Alternativ wäre für einen prototypischen Ansatz auch die Nutzung von SQLite denkbar, da dieses System einfach zu implementieren und zu konfigurieren ist. 
Aufgrund der eingeschränkten Persistenz lokaler Daten in einer Cloud-basierten Umgebung wurde jedoch bewusst auf SQLite verzichtet.

\subsection{Infrastruktur}
Die Infrastruktur des Systems ist in drei Umgebungen unterteilt: Entwicklungs-, Test- und Produktivumgebung. 
Die Entwicklung erfolgt auf der lokalen Entwicklermaschine. 
Die Test- sowie eine mögliche Produktivumgebung stellt der Cloud-Hosting-Betreiber \textit{Laravel Cloud} zur Verfügung \parencite{laravelCloud}.
Es wird der Cloud Service \textit{Platform as a Service}, kurz PaaS, von Laravel Cloud genutzt, da somit das Hauptaugenmerk auf die Entwicklung gelegt werden kann und nicht auf die Infrastruktur.
Beim Betreiben von Websites ist heutzutage Cybersecurity ein extrem wichtiger Aspekt. 
Da in der ATS-Applikation sensible Daten gespeichert werden, ist hier höchste Vorsicht geboten. 
Die Absicherung der Infrastruktur, beispielsweise das Abfangen von DDoS-Attacken, obliegt somit dem Cloud-Anbieter. 
Würde man hingegen einen eigenen Webserver betreiben, müsste man sich um die Security selbst kümmern.
Laravel Cloud wurde speziell für Laravel-Anwendungen entwickelt und bietet native Unterstützung für PostgreSQL-Datenbanken \parencite{laravelCloudDocs}. 
Dies ist auch der Grund für die Nutzung von PostgreSQL als Datenbanksystem.
Wie bei Cloud-Hosting üblich, gibt es verschiedene Preismodelle. 
Laravel Cloud verfolgt ein nutzungsbasiertes Abrechnungsmodell ohne feste monatliche Grundgebühr \parencite{laravelCloudPricing}. 
Die Kosten entstehen ausschließlich durch aktive Nutzung. 
Aufgrund von Auto-Hibernation skaliert die Anwendung bei Inaktivität automatisch auf null herunter, wodurch keine Kosten für Leerlaufzeiten anfallen. 
Für den MVP wird der \textit{Starter Plan} genutzt, welcher folgende Leistungen umfasst:
\begin{itemize}
\item App-Hosting - Starter Plan (Flex Compute, mit Hibernation) \parencite{laravelCloudPricing}
\item Datenbank-Hosting - Serverless PostgreSQL (mit Hibernation) \parencite{laravelCloudPricing}
\item Monatliche Kosten - nutzungsabhängig, ca. 1--2 USD bei geringer Aktivität
\end{itemize}

\subsection{Softwarearchitektur}
Für die Umsetzung der Applikation wurde die Architektur nach dem Model-View-Controller (MVC)-Prinzip gewählt. 
Diese unterteilt das System in drei zentrale Komponenten: 
Das Model umfasst die Datenhaltung sowie die Geschäftslogik, die View ist für die Darstellung der Benutzeroberfläche verantwortlich und der Controller fungiert als Vermittler zwischen Model und View \parencite{mvc}.
\newline
Diese Architektur eignet sich insbesondere für kleinere bis mittelgroße Webanwendungen und wird vom verwendeten Framework Laravel standardmäßig unterstützt. 
Dadurch kann eine strukturierte und zugleich schlanke Umsetzung des Systems gewährleistet werden, die dem Umfang eines Minimum Viable Product entspricht.
\newline
Für größere und komplexere Anwendungen kommen hingegen häufig Architekturmuster wie Model-View-ViewModel (MVVM) oder Microservice-Architekturen zum Einsatz, die eine höhere Skalierbarkeit und Flexibilität ermöglichen.
\newline
Im Rahmen der Architekturdefinition wurden zudem zentrale Qualitätsmerkmale identifiziert und berücksichtigt. 
Dazu zählen insbesondere die Performance, die Wartbarkeit sowie die Sicherheit der Anwendung (vgl. Tabellen \ref{tab:performance}, \ref{tab:wartbarkeit} und \ref{tab:sicherheit}).
